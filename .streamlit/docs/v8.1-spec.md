# V8.1 Spec — Hiện trạng sau khắc phục V8.3 & Đề xuất cải tiến thêm

Tài liệu này dựa trên **v8-spec**, cập nhật theo hiện trạng sau khi áp dụng các khắc phục (V8.3), bổ sung mục **những điểm đã khắc phục và cách khắc phục**, và giữ **đề xuất cải tiến thêm** (những ý mới).

---

## 1. Flow gọi LLM hiện tại vs flow không dùng LLM

### 1.1. Các flow **có gọi LLM** (sau V8.3)

| Vị trí | Mục đích | Ghi chú |
|--------|----------|--------|
| **Router / Planner (3 bước thống nhất)** | | |
| `SmartAIRouter.intent_only_classifier` | Phân loại intent (chung Router & Planner) | 1 LLM, tính vào giới hạn turn |
| `SmartAIRouter.context_planner` | Khi `needs_data` (single intent): context_needs, chapter_range... | 1 LLM, tính vào giới hạn |
| `SmartAIRouter.get_plan_v7_light` | Plan tối đa 3 bước (khi multi-intent / suggest_v7) | 1 LLM, tính vào giới hạn; prompt ngắn |
| `SmartAIRouter.ai_router_pro_v2` | Fallback một lần (khi plan lỗi/rỗng) | 1 LLM |
| **Chat — trả lời chính** | | |
| `web_search` / `chat_casual` | Build context (web/search hoặc persona) → LLM trả lời | 1 LLM, tính vào giới hạn |
| `search_context` | `build_context` (V8 full: chapter, bible, relation, timeline, chunk + bổ sung toàn văn chương khi thiếu) → LLM trả lời | 1 LLM (+ optional is_answer_sufficient, fallback retry); tính 1 vào giới hạn |
| `query_Sql`, `check_chapter_logic`, `update_data` (confirm) | Như v8-spec | 1–2 LLM, tính vào giới hạn |
| `numerical_calculation` | Build context → LLM sinh code → executor → LLM trả lời | 2 LLM, cả hai tính vào giới hạn (trong turn/executor có `llm_budget_ref`) |
| **V7 execute_plan** | build_context từng step; numerical: LLM code (nếu còn budget) | Chỉ numerical tính vào giới hạn |
| Draft sau execute_plan | 1 LLM từ cumulative context | Tính vào giới hạn |
| **Verification / check (không tính vào giới hạn)** | | |
| `run_verification_loop` | `verify_output` → có thể `_verify_grounding_llm`; correction LLM | **Không** tính vào max_llm_calls_per_turn |
| `_verify_grounding_llm` | LLM-as-judge (đã tối ưu: slice 6k/2.5k, max_tokens 150) | Không tính |
| `is_answer_sufficient` | Heuristic hoặc LLM sufficient true/false | Không tính |
| **Background / khác** | Unified, chapter_logic_check, persona extractor, ... | Như v8-spec |

### 1.2. Các flow **không dùng LLM** để tạo câu trả lời chính

Giữ nguyên như v8-spec: ask_user_clarification, suggest_v7, chỉ lệnh @@ fallback, build context (chỉ DB/API), query_Sql phần lấy dữ liệu.

## 2. Các loại intent và cách code thực thi (sau V8.3)

- Bảng tóm tắt intent giữ như v8-spec; bổ sung:
  - **search_context:** Luôn gather đủ (FULL_CONTEXT_NEEDS_V8) khi bật V8 full; có thể tắt bằng settings `v8_full_context_search`.
  - **Planner path:** Dùng chung bước 1 intent_only; bước 2 là get_plan_v7_light (multi) hoặc context_planner (single); bước 3 execute + draft. Có giới hạn LLM/turn và `llm_budget_ref` cho numerical.
- **INTENT_HANDLER_MAP** trong `ai/router.py`: intent → handler (clarification, template, llm_casual, llm_with_context, data_operation) dùng tham chiếu thống nhất.

---

## 3. Context dùng để trả lời câu hỏi user (sau V8.3)

- Thành phần chung: giữ như v8-spec (persona, arc, strict, rules).
- **search_context (V8 full):**
  - Khi bật `v8_full_context_search`: `context_needs` và `context_priority` = `FULL_CONTEXT_NEEDS_V8` (chapter, bible, relation, timeline, chunk); thứ tự thu thập theo đó.
  - Chunk: `top_k=20`, reverse lookup `token_limit=6000`; timeline `limit=30`.
  - Sau khi gather, nếu `total_tokens < 35% max_context_tokens` và có chapter_range: append block **toàn văn chương** (nhãn V8 bổ sung khi context còn thiếu).
- Observability: mỗi turn (nhánh có context) gọi `log_chat_turn(...)` ghi vào `chat_turn_logs` (intent, context_needs, context_tokens, llm_calls_count).

## 4. Những điểm đã khắc phục và cách khắc phục

### 4.1. Giới hạn số lần gọi LLM mỗi turn

- **Vấn đề (v8-spec):** Một lượt chat có thể gọi nhiều LLM (intent + planner + trả lời + sufficient + fallback + verifier), tốn chi phí và latency; chưa có cơ chế giới hạn.
- **Cách khắc phục:**
  - Thêm cấu hình **max_llm_calls_per_turn** (Settings → V8 & Observability), mặc định **5**, giá trị **0** = không giới hạn.
  - Đếm chỉ các gọi “chính”: intent_only, context_planner, get_plan_v7_light, draft, numerical code gen. **Verification và check (verify_output, is_answer_sufficient) không tính** vào giới hạn.
  - Trước mỗi gọi tính vào giới hạn: kiểm tra `llm_calls_this_turn[0] < max_llm_calls_per_turn`; khi đạt giới hạn thì không gọi thêm, hiển thị thông báo và gợi ý tăng trong Settings.
  - **File:** `config.py` (get_max_llm_calls_per_turn), `views/settings.py` (number input + lưu), `views/chat.py` (biến đếm, kiểm tra trước mỗi LLM), `core/executor_v7.py` (llm_budget_ref cho numerical).

### 4.2. Planner lag và tách rõ Router / Planner

- **Vấn đề (v8-spec):** Cần tách rõ router path và V7 path; tránh gọi dồn intent_only + context_planner + ai_router_pro_v2; Planner (get_plan_v7) prompt nặng, dễ lag.
- **Cách khắc phục:**
  - **Planner theo mô hình 3 bước giống Router:**
    - **Bước 1:** Luôn gọi **intent_only_classifier** (1 LLM) chung cho cả Router và Planner.
    - **Bước 2:** Nếu bật V7 Planner và (intent = suggest_v7 hoặc `is_multi_intent_request(prompt)`): gọi **get_plan_v7_light** (prompt ngắn, tối đa 3 bước, max_tokens 500). Nếu không: single intent → context_planner khi needs_data, không thì dùng luôn router_out từ step1.
    - **Bước 3:** Execute (chỉ build context, không LLM) + 1 LLM draft. Numerical trong execute_plan chỉ gọi LLM khi còn budget (`llm_budget_ref`).
  - Thêm **get_plan_v7_light** trong `ai/router.py`: prompt rút gọn, plan tối đa 3 bước; lỗi/rỗng thì fallback `_single_intent_to_plan(ai_router_pro_v2(...))`.
  - **File:** `views/chat.py` (luồng thống nhất intent_only → plan hoặc context_planner → execute/answer), `ai/router.py` (get_plan_v7_light).

### 4.3. Search context thiếu ý (chỉ một loại search)

- **Vấn đề (v8-spec):** search_context phụ thuộc context_needs/context_priority từ router, dễ chỉ lấy một loại (chapter hoặc bible) → trả lời thiếu ý.
- **Cách khắc phục:**
  - Khi intent **search_context** và bật **V8 full** (mặc định): override `context_needs` và `context_priority` thành **FULL_CONTEXT_NEEDS_V8** (chapter, bible, relation, timeline, chunk); thu thập đủ tất cả, không chỉ một loại.
  - Tăng chunk (top_k 20, token_limit 6000) và timeline (limit 30); thêm block **bổ sung toàn văn chương** khi tổng token sau gather vẫn < 35% max_context_tokens.
  - Feature flag **v8_full_context_search** (Settings): tắt thì dùng lại context_needs/context_priority từ router.
  - **File:** `ai/context_schema.py` (FULL_CONTEXT_NEEDS_V8), `ai_engine.py` (override, bổ sung chương, đọc flag).

### 4.4. Verification/check tốn token và chậm

- **Vấn đề:** Verification (LLM judge) dùng context/response dài, max_tokens lớn.
- **Cách khắc phục:**
  - **Tối ưu _verify_grounding_llm:** `max_context_chars=6000`, response slice 2500 ký tự, `max_tokens=150` (trước 10000, 4000, 200). Verification/check **không** tính vào giới hạn LLM/turn.
  - **File:** `ai_verifier.py`.

### 4.5. Numerical trong execute_plan không tôn trọng giới hạn turn

- **Vấn đề:** Mỗi step numerical gọi LLM, có thể vượt giới hạn turn.
- **Cách khắc phục:**
  - Thêm tham số **llm_budget_ref: Optional[List[int]]** = [current_count, max_count] vào **execute_plan**. Chỉ gọi LLM cho numerical_calculation khi `current < max`; gọi xong tăng current. Hết budget: không gọi, ghi "(Bỏ qua: đã đạt giới hạn gọi LLM cho lượt này.)" vào context.
  - Chat truyền `llm_budget_ref=llm_calls_this_turn + [max_llm_calls_per_turn]` khi gọi execute_plan.
  - **File:** `core/executor_v7.py`, `views/chat.py`.

### 4.6. Thiếu observability và feature flags

- **Vấn đề (v8-spec):** Cần log intent, token_count, số lần LLM; cần feature flags để A/B test và rollback.
- **Cách khắc phục:**
  - **Observability:** Bảng **chat_turn_logs** (migration V8.3); module **core/observability.py** (`log_chat_turn`); gọi từ chat trước khi gọi LLM trả lời (intent, context_needs, context_tokens, llm_calls_count).
  - **Feature flags / Settings:** Tab **Settings → V8 & Observability**: toggle V8 full context search, number input max_llm_calls_per_turn; lưu vào bảng `settings`.
  - **File:** `.streamlit/data/schema_v8_3_migration.sql`, `core/observability.py`, `views/chat.py`, `views/settings.py`, `config.py`.

### 4.7. Chuẩn hóa intent → handler (v8-spec 4.2)

- **Vấn đề:** if/elif rải rác nhiều chỗ theo intent.
- **Cách khắc phục:**
  - Thêm **INTENT_HANDLER_MAP** trong `ai/router.py`: intent → handler type (clarification, template, llm_casual, llm_with_context, data_operation). Dùng tham chiếu thống nhất; chat/executor vẫn dùng intent trực tiếp.
  - **File:** `ai/router.py`.

## 5. Đề xuất cải tiến thêm (những ý mới)

### 5.1. Flow LLM và chi phí

- **Cache intent/planner:** Cache kết quả intent_only (và context_planner hoặc get_plan_v7_light) theo (project_id, hash(prompt)) với TTL ngắn (vài chục giây) để tránh gọi LLM lặp khi user gửi lại gần giống.
- **Tùy chọn tắt verification theo intent:** Cho phép cấu hình “không verify” với một số intent (ví dụ chat_casual, web_search) để giảm latency khi không cần grounding chặt.

### 5.2. Intent và thực thi

- **Refactor dùng INTENT_HANDLER_MAP:** Thay if/elif theo intent bằng dispatch theo handler (clarification, template, llm_casual, llm_with_context, data_operation) để code gọn và dễ mở rộng.
- **Unit test query_Sql vs search_context:** Thêm test với vài câu mẫu để đảm bảo router phân biệt rõ “xem/liệt kê dữ liệu thô” (query_Sql) và “hỏi tự nhiên” (search_context).
- **Numerical — kiểm tra code an toàn:** Trước khi PythonExecutor.execute, thêm bước allowlist thư viện / sandbox (nếu có) để giảm rủi ro.
- **check_chapter_logic — soát theo dimension:** Cho phép user chọn “chỉ soát dimension X” (timeline, bible, relation, chat_crystallize, rule) để giảm prompt và token.
- **update_data (unified):** Option “chỉ extract bible” hoặc “chỉ timeline” để tiết kiệm token khi không cần đủ 6 loại output.

### 5.3. Context build

- **Cap token rõ theo từng need:** Ghi rõ trong spec (hoặc config) cap tối đa token cho từng need (chapter, bible, timeline, chunk) khi có max_context_tokens.
- **Dedupe context:** Bible + chapter có thể trùng đoạn; cân nhắc dedupe theo paragraph hash hoặc cắt bớt khi sắp vượt token.
- **Lỗi từng phần:** Nếu một phần (vd timeline) lỗi hoặc trống, vẫn tiếp tục thêm bible/chapter và có thể thêm dòng “[TIMELINE] Lỗi tải” để LLM không bịa.
- **Metrics context_priority vs token:** Log hoặc metrics “context_priority vs token thực tế” để tinh chỉnh mặc định theo loại câu hỏi.

### 5.4. Phiên bản và vận hành

- **context_version trong router output:** Nếu sau này đổi format router (vd context_version: 2), build_context đọc version và xử lý tương thích ngược.
- **Feature flags mở rộng:** Bật/tắt từng bước: chỉ intent (không planner), chỉ V2 một LLM, tắt is_answer_sufficient, tắt fallback full chapter, v.v.
- **UI xem log observability:** Tab hoặc trang xem bảng chat_turn_logs (lọc theo project, user, ngày) để debug và tối ưu chi phí.
