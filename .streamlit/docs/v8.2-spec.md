## 2. Các loại intent và cách code thực thi (sau V8.2)

- Bảng tóm tắt intent giữ như v8.1.
- **V8.2 — Dispatch theo handler:** Trong `ai_engine.py`, `ContextManager.build_context` **không còn if/elif theo từng intent**. Thay bằng:
  - `handler_type = INTENT_HANDLER_MAP.get(intent, "llm_with_context")`
  - Gọi `INTENT_CONTEXT_HANDLERS[handler_type](router_result, ctx)` với `ctx` chứa context_parts, sources, total_tokens, project_id, session_state, context_needs, context_priority, ...
  - Các handler: **clarification**, **template**, **llm_casual**, **data_operation**, **llm_with_context** (gồm query_Sql, check_chapter_logic, search_context/numerical gathering).
- **check_chapter_logic:** Hỗ trợ tham số `dimensions` (list hoặc None). Tab Data Health cho phép chọn "All" hoặc từng dimension (timeline, bible, relation, chat_crystallize, rule) để giảm prompt và token.

## 5. Đã thực hiện trong V8.2 (cải tiến từ đề xuất v8.1)

### 5.1. Refactor dùng INTENT_HANDLER_MAP trong build_context

- **Mục tiêu:** Thay if/elif theo intent bằng dispatch theo handler để code gọn và dễ mở rộng.
- **Cách làm:**
  - Trong `ai_engine.py`: Thêm các hàm handler `_intent_handle_clarification`, `_intent_handle_template`, `_intent_handle_llm_casual`, `_intent_handle_data_operation`, `_intent_handle_llm_with_context` và map `INTENT_CONTEXT_HANDLERS`.
  - `build_context` sau khi chuẩn hóa intent và context_needs/context_priority: tạo `ctx` (context_parts, sources, total_tokens, project_id, session_state, ...), gọi `handler_fn(router_result, ctx)` với `handler_type = INTENT_HANDLER_MAP.get(intent, "llm_with_context")`.
  - Handler **llm_with_context** xử lý: query_Sql (và fallback search_context nếu không có block), check_chapter_logic (có truyền dimensions từ ctx nếu có), sau đó full search_context/numerical gathering.
- **File:** `ai_engine.py` (handler functions + dispatch), `ai/router.py` (export INTENT_HANDLER_MAP).

### 5.2. Unit test query_Sql vs search_context

- **Mục tiêu:** Đảm bảo router phân biệt rõ “xem/liệt kê dữ liệu thô” (query_Sql) và “hỏi tự nhiên” (search_context).
- **Cách làm:**
  - Thêm `tests/test_router_query_sql_vs_search_context.py`: mock `AIService.call_openrouter` trả về JSON intent cố định; gọi `SmartAIRouter.intent_only_classifier` với câu mẫu; assert intent = query_Sql cho mẫu “liệt kê chương” / “xem danh sách timeline”, intent = search_context cho mẫu “nhân vật A làm gì” / “quan hệ B và C”.
  - Các hằng mẫu: `QUERY_SQL_SAMPLES`, `SEARCH_CONTEXT_SAMPLES`.
  - Test thêm: INTENT_HANDLER_MAP có query_Sql và search_context map tới llm_with_context.
- **Chạy test:** Trong môi trường có streamlit và dependency: `python -m pytest tests/test_router_query_sql_vs_search_context.py -v` hoặc `python -m unittest tests.test_router_query_sql_vs_search_context -v`.
- **File:** `tests/test_router_query_sql_vs_search_context.py`.

### 5.3. Numerical — kiểm tra code an toàn (PythonExecutor sandbox)

- **Mục tiêu:** Trước khi PythonExecutor.execute, thêm bước allowlist thư viện / cấm từ khóa nguy hiểm để giảm rủi ro.
- **Cách làm:**
  - Trong `utils/python_executor.py`: Hàm **validate_code_safety(code)** kiểm tra:
    - **Cấm:** `__import__`, `open(`, `eval(`, `exec(`, `compile(`, `breakpoint(`, `input(`, `globals(`, `locals(`, `vars(`, `getattr(`, `setattr(`, `delattr(`, `__builtins__`, `os`, `subprocess`, `socket`, `sys`.
    - **Import:** Chỉ cho phép `import math`, `import numpy`/`np`, `import pandas`/`pd`, `import json` (và `from ... import ...` tương ứng).
  - `PythonExecutor.execute()` gọi `validate_code_safety(code)` trước `exec()`; nếu không qua thì trả về `(None, "Sandbox: ...")`.
  - Unit test: `tests/test_python_executor_safety.py` (safe pandas/numpy pass; open/eval/exec bị chặn; execute từ chối code không an toàn).
- **File:** `utils/python_executor.py`, `tests/test_python_executor_safety.py`.

### 5.4. check_chapter_logic — soát theo dimension

- **Mục tiêu:** Cho phép user chọn “chỉ soát dimension X” (timeline, bible, relation, chat_crystallize, rule) để giảm prompt và token.
- **Cách làm:**
  - **core/chapter_logic_check.py:**
    - `build_logic_context_for_chapter(..., dimensions=None)`: Nếu `dimensions` là list (vd. `["timeline", "bible"]`), chỉ đưa vào context các phần tương ứng; None hoặc rỗng = đủ 5 dimension.
    - `run_chapter_logic_check(..., dimensions=None)`: Gọi build_logic_context_for_chapter với dimensions; prompt chỉ yêu cầu soát đúng các dimension đã chọn.
  - **views/data_health.py:** Trong tab Data Health, thêm multiselect **“Dimensions cần soát”** với options `All` + timeline, bible, relation, chat_crystallize, rule. Mặc định “All” → `dimensions=None`; chọn một vài mục → truyền list tương ứng vào `run_chapter_logic_check`.
  - **ai_engine.py:** Trong handler llm_with_context, gọi `run_chapter_logic_check(..., dimensions=ctx.get("logic_check_dimensions"))` (từ chat vẫn None = full).
- **File:** `core/chapter_logic_check.py`, `views/data_health.py`, `ai_engine.py`.

## 6. Đề xuất cải tiến thêm (chưa làm)

*(Các ý còn lại từ v8.1 section 5, và mở rộng.)*

### 6.1. Flow LLM và chi phí

- Cache intent/planner theo (project_id, hash(prompt)) với TTL ngắn.
- Tùy chọn tắt verification theo intent (chat_casual, web_search).

### 6.2. Intent và thực thi

- **update_data (unified):** Option “chỉ extract bible” hoặc “chỉ timeline” để tiết kiệm token.

### 6.3. Context build

- Cap token rõ theo từng need (chapter, bible, timeline, chunk).
- Dedupe context (bible + chapter trùng đoạn).
- Lỗi từng phần: vẫn tiếp tục thêm phần khác, ghi rõ “[TIMELINE] Lỗi tải” nếu cần.
- Metrics context_priority vs token.

### 6.4. Phiên bản và vận hành

- context_version trong router output (tương thích ngược).
- Feature flags mở rộng (chỉ intent, tắt is_answer_sufficient, tắt fallback full chapter, …).
- UI xem log observability (chat_turn_logs).
